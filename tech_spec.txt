1. What This Tool Does
Every Corgi ETF must file a monthly portfolio holdings report (Form N-PORT) with the SEC. The filing is a structured XML document containing every holding, fund financials, risk metrics, and securities lending data.
This tool takes a canonical set of fund parameters (registrant identity, filing-period financials, and a list of holdings) as input, converts them to SEC-compliant N-PORT XML, validates the result against the SEC's XSD schema, and submits it to EDGAR via the SEC's Submission API.
The canonical input schema is defined and owned by us — it is intentionally decoupled from any single data source. A US Bank Excel file is one upstream source we will adapt into canonical inputs; it is not the tool's contract. Other sources (a different administrator, our own books and records, a PMS export, hand-authored test fixtures, or amendments reconstructed after the fact) feed the same canonical inputs through their own adapters.
We currently outsource this to a third-party vendor. We're bringing it in-house because we're scaling from 2 funds to 300+, and per-fund vendor costs don't scale.

2. Functional Requirements
The tool MUST:
1. Accept canonical fund inputs — FundConfig (registrant identity, series/class IDs, signer, EDGAR credentials), FilingData (period, NAV, balance sheet, returns, flows, risk metrics), and Holdings (one record per position) — in a stable, source-agnostic schema we own
2. Generate XML conforming to the SEC's N-PORT XSD schema (v1.13) byte-for-byte equivalent (modulo whitespace) to a known-good reference filing when fed the same canonical inputs
3. Validate the XML against the XSD locally before submission
4. Auto-determine submission type: NPORT-P (public, quarter-end) or NPORT-NP (non-public, other months)
5. Submit to EDGAR via the Submission API using JWE auth tokens
6. Track filing status (accepted vs. suspended) via the EDGAR Status API
7. Support amendments (NPORT-P/A, NPORT-NP/A)
8. Run in batch across 100-337 funds in a single execution
9. Archive every XML with hash, log every submission with accession number and status
10. Support test mode (EDGAR Beta environment, liveTestFlag=TEST)
The tool SHOULD:
11. Provide source-specific adapters (US Bank xlsx, future BNY xlsx, JSON, database) that produce canonical inputs — kept strictly separate from the XML generator
12. Be schema-version-aware (swap XSD files without code changes)
13. Manage a shared counterparty registry (name + LEI) across funds

The tool auto-determines this from fiscal year end + reporting period:
Month 3 of fiscal quarter = NPORT-P (public)
Month 1 or 2 = NPORT-NP (non-public)
Monthly Timeline

Day 20-25: Human spot-check (optional)

8.3 Leveraged 2x Daily Fund (Direxion TSLA Bull 2X, $5.8B AUM)
Source: Direxion TSLA 2X, Jan 2026, accession 0001193125-26-137760. 13 holdings: 7 total return swaps + 1 TSLA equity position + 5 money market funds.
What's different: Each swap is its own invstOrSec with a swapDeriv block inside derivativeInfo. Key patterns:
name = N/A, lei = N/A, cusip = N/A (swaps don't have standard identifiers)

payoffProfile = N/A
assetCat = DE (derivative-equity)

8.4 Buffer / Defined Outcome Fund (FT Vest Buffer ETF, $199M AUM)
Source: FT Vest U.S. Equity Enhance & Moderate Buffer ETF, Nov 2025, accession 0000940400-26-002430. 16 holdings: 15 FLEX options + 1 money market fund.

1. Purchased Call (deep ITM) -- provides upside participation
2. Written Call (OTM) -- caps the upside ("cap")
3. Purchased Put (ATM) -- provides downside buffer
4. Written Put (OTM) -- defines the buffer floor
